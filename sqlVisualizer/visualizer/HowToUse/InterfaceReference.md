# Interface Reference

This document describes the interfaces used across the SQL Visualizer solution, what each interface represents, where it belongs in the architecture, and which concrete implementation is currently used.

## End-to-end pipeline

At a high level, the application processes a query in this order:

1. The UI collects SQL and selects the active database.
2. `ISQLInputValidator` validates the SQL.
3. `ISQLParser` decomposes the query into clause-level components.
4. `ITablesPerExecutionStepGenerator` generates intermediate execution steps.
5. `IAnimationGenerator` converts those execution steps into visual animation steps.
6. The Blazor UI renders `IDisplayTable` and `IAnimation` instances.
7. `IMetricsHandler` and `IUserRepository` persist usage data and per-session state.

## `common.data`

### `ISimpleTable`

File: [ISimpleTable.cs](C:/My%20stuff/Uni/Uni-stuff-git/6%20Semester/Bachelor/Itu-bsc-2026-sqlVisualizer/sqlVisualizer/common.data/ISimpleTable.cs)

Purpose:
Provides the smallest shared table shape in the system: column names plus raw rows.

Members:
- `IList<string> ColumnNames()`: Returns the column names in display/execution order.
- `IList<IList<object?>> Rows()`: Returns raw row data. Each inner list is one row.

Used by:
- `ISQLExecutor.Execute(...)`
- `IDisplayTableGenerator.Generate(...)`
- Database inspection and query execution paths

Current implementations:
- `sql.executor.Models.Table`

Notes:
- This is the neutral tabular contract used before the data becomes visualization-specific.

### `IDatabaseTable`

File: [IDatabaseTable.cs](C:/My%20stuff/Uni/Uni-stuff-git/6%20Semester/Bachelor/Itu-bsc-2026-sqlVisualizer/sqlVisualizer/common.data/IDatabaseTable.cs)

Purpose:
Represents one physical/logical table inside a browsable database schema.

Members:
- Inherits `ISimpleTable`
- `string Name()`: Table name
- `IReadOnlyList<string> ColumnTypes()`: Column type names aligned with `ColumnNames()`

Used by:
- `IDatabase.Tables()`
- `SchemaView` when rendering the schema sidebar

Current implementations:
- `sql.executor.Models.Table`

Notes:
- This interface extends `ISimpleTable` because schema browsing and raw table preview use the same row/column structure.

### `ISQLExecutor`

File: [ISQLExecutor.cs](C:/My%20stuff/Uni/Uni-stuff-git/6%20Semester/Bachelor/Itu-bsc-2026-sqlVisualizer/sqlVisualizer/common.data/ISQLExecutor.cs)

Purpose:
Abstracts SQL execution against the currently selected DuckDB database.

Members:
- `Task<ISimpleTable> Execute(string sqlQuery)`: Executes a SQL query and returns a raw result table

- `SchemaView`
- `SQLExecutorWrapper`
- The standalone `/Editor` page

Current implementations:
- `sql.executor.SQLExecutor`

Notes:
- `Execute(...)` is used both for actual user queries and for internal queries generated by the visualization pipeline.

## Practical dependency map

If you want to understand where a change belongs, this is the shortest route:

- Change database selection or query execution:
  `ICurrentDatabaseContext` -> `ISQLExecutor` -> `SchemaView` / `/Editor`

- Change parsing/decomposition:
  `ISQLInputValidator` -> `ISQLParser` -> `ISQLDecompositionComponent`

- Change execution-step generation:
  `ITablesPerExecutionStepGenerator` -> `ISqlExecutedStep` -> `ITable`

- Change animation behavior:
  `IAnimationGenerator` -> `IAnimation` -> `IDisplayTable`

- Change rendered table behavior:
  `IDisplayTable`, `IDisplayTableRow`, `IDisplayTableCell`, `IDisplayAggregation`

- Change persistence/analytics:
  `IUserRepository`, `IMetricsHandler`

## Current gaps worth knowing

- `IDisplayAggregation.HexBackgroundColor()` is declared but not fully implemented in the current concrete class.
- `OFFSET` exists in the SQL model but does not yet have a visualization implementation.
- Some interfaces are defined close to their implementation rather than in one centralized contracts package, so this document is useful as the canonical overview.
